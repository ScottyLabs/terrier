# RFC 0009: ScottyLabs Access Control (SLAC)

- **Status:** Accepted
- **Author(s):** @kritdass
- **Created:** 2026-04-18
- **Updated:** 2026-04-18

## Overview

This RFC establishes a small, domain-free Rust crate, `crates/slac`, that lifts authorization out of axum handler bodies and into the type system. Handlers declare their access requirements as a type parameter, and the framework runs the check before the handler body executes.

## Motivation

Terrier is built on Axum and authenticates users through Keycloak (with `saml-proxy` from RFC 0005 sitting in front for university SSO). Per RFC 0008, per-hackathon roles live in `user_hackathon_roles`. Access control (given an authenticated user, deciding whether they may do a particular thing) is the missing layer between the OIDC stack and each handler.

Without a shared mechanism, every handler ends up performing the same load-resource-then-check pattern in its body. The auth contract becomes **invisible from the handler signature**, **easy to forget** with no compile-time signal, and **repeated across dozens of routes**. Every ScottyLabs Rust service will hit the same problem, so we should solve it once.

## Goals

- Make authorization requirements part of every handler's type signature
- Make forgetting an auth check a compile-time error
- Keep the core crate domain-free so it can be reused across services
- Compose with the existing Keycloak + `axum-oidc` stack, not replace it
- Stay small and stable enough for cross-project dependence

## Non-Goals

- Replacing or modifying the authentication layer (Keycloak owns identity)
- Filtering rows inside SQL queries (handler-level authorization only)
- Providing a permission administration UI (assignment lives in the database per RFC 0008)
- Building a procedural macro framework (only a `macro_rules!` for sugar)

## Detailed Design

### Before and After

The same endpoint (letting a hackathon admin delete a team) written both ways:

```rust
pub async fn delete_team(
    user: CurrentUser,
    Path((slug, team_id)): Path<(String, i32)>,
    State(s): State<AppState>,
) -> Result<StatusCode, ApiError> {
    let hackathon = Hackathons::find()
        .filter(hackathons::Column::Slug.eq(&slug))
        .one(&s.db).await?
        .ok_or(ApiError::NotFound)?;

    let role = UserHackathonRoles::find()
        .filter(user_hackathon_roles::Column::UserId.eq(user.id))
        .filter(user_hackathon_roles::Column::HackathonId.eq(hackathon.id))
        .one(&s.db).await?
        .ok_or(ApiError::Forbidden)?;

    if !s.config.global_admins.contains(&user.email) && role.role != "admin" {
        return Err(ApiError::Forbidden);
    }

    Teams::delete_by_id(team_id).exec(&s.db).await?;
    Ok(StatusCode::NO_CONTENT)
}
```

With the library, the lookup and the check live inside `IsHackathonAdmin` and are reused by every handler that needs the same guarantee:

```rust
pub async fn delete_team(
    Authorized { data, .. }: Authorized<IsHackathonAdmin>,
    Path((_, team_id)): Path<(String, i32)>,
) -> Result<StatusCode, ApiError> {
    let (_user, hackathon) = data;
    Teams::delete_by_id(team_id).exec_in(&hackathon).await?;
    Ok(StatusCode::NO_CONTENT)
}
```

Removing the extractor either breaks the body (no `data` to destructure) or reduces the handler to something a reviewer would immediately reject. The loaded `hackathon` flows in as part of the proof, so the body avoids a second DB round-trip.

### Three Concepts

A **`Policy`** is a trait whose `check` method takes the request parts and app state and either produces an `Output` value (the resource the handler will use) or rejects with an `Error` (an HTTP response). An **`Authorized<P>`** is a witness type whose only constructor is its `FromRequestParts` implementation, which calls `P::check`. The **`policy!`** macro builds a sum type from several `Policy` impls, with each variant carrying that policy's `Output`. This is useful when an endpoint is reachable by multiple tiers and the handler matches on which one applied.

```rust
pub trait Policy<S>: Sized + Send + Sync + 'static {
    type Output: Send + Sync;
    type Error: IntoResponse + Send;
    fn check(parts: &mut Parts, state: &S)
        -> impl Future<Output = Result<Self::Output, Self::Error>> + Send;
}

pub struct Authorized<P: Policy<S>, S = ()> {
    pub data: P::Output,
    _proof: PhantomData<P>, // private => cannot forge
}

policy! {
    pub enum SettingsAccess {
        GlobalAdmin    = IsGlobalAdmin,    // tried first
        HackathonAdmin = IsHackathonAdmin,
    }
}
```

Adding a new tier to `SettingsAccess` later (e.g. `Auditor`) makes every existing `match` non-exhaustive and produces a compile error at each call site, forcing the new role to be considered explicitly.

Most policies are reused across many handlers and warrant a named struct. For one-off checks (something only one handler will ever need), defining a struct is overkill; `from_fn` accepts a closure with the same shape as `Policy::check` and returns a `Policy` value:

```rust
let owns_post = from_fn(|parts, state: &AppState| async move {
    let user = CurrentUser::from_request_parts(parts, state).await?;
    let Path(post_id) = Path::<i32>::from_request_parts(parts, state).await?;
    let post = Posts::find_by_id(post_id).one(&state.db).await?
        .ok_or(ApiError::NotFound)?;
    if post.author_id == user.id { Ok((user, post)) }
    else { Err(ApiError::Forbidden) }
});
```

This is functionally identical to writing `struct OwnsPost; impl Policy for OwnsPost { ... }`, but stays inline at the call site. The combinators `Any<A, B>` and `All<A, B>` compose policies (named or inline) into "either A or B" and "both A and B" respectively.

### Integration with the OIDC Stack

SLAC does not replace anything beneath it. The flow is:

1. **Keycloak** issues an OIDC token after authentication (with `saml-proxy` brokering university IdPs upstream per RFC 0005).
1. **`axum-oidc`** validates the JWT on every protected route.
1. A small middleware in `terrier-server` upserts the user row into `users` and inserts a `CurrentUser` value into the request extensions.
1. Each `Policy` implementation extracts `CurrentUser`, reads any relevant route parameters or rows from the entities defined in RFC 0008, and either returns the resource or rejects.

The Keycloak global-admin override (configured via environment per RFC 0001) lives once per policy that needs it, instead of being restated in every handler body.

### Workspace Placement

The crate is added to the workspace as `crates/slac`, alongside the existing `crates/saml-proxy`, `crates/terrier-common`, and `crates/terrier-server`. Terrier-specific atomic policies (`IsHackathonAdmin`, `IsJudge`, etc.) live in `crates/terrier-server/src/auth/`, keeping SLAC itself free of any Terrier types.

## Alternatives Considered

### Tower middleware per route

Wrapping each route in `.layer(require_admin())` keeps the auth contract in the router rather than the handler signature, requires sharing fetched data through type-erased extensions, and turns a missing layer into a runtime bug rather than a compile error.

### Procedural macro attribute

A `#[require(Admin)]` attribute on each handler is opaque to rust-analyzer, can be silently present on a handler that never reads user data, and obligates us to maintain a procedural macro indefinitely.

### Keycloak Authorization Services (UMA)

Pushing per-request decisions to Keycloak adds a remote round-trip on every protected route, splits the source of truth between code and Keycloak's UI, and provides no path for handing the loaded resource into the handler. Role *claims* from Keycloak can still be consumed via a small `Policy` if we ever need realm-level checks.

### Hand-rolling per service

If three or four ScottyLabs services each invent their own auth helpers, we end up with three or four divergent solutions to audit and maintain. A small shared crate is strictly cheaper.

## Open Questions

1. Whether to provide a recommended `AuthError` enum or stay fully generic via the `type Error` associated type. The crate will ship generic with a documented example until consumer needs converge.
1. Whether to provide a helper that keeps the `utoipa-axum` `security(...)` annotation in sync with the policy in the handler signature. Deferred until we observe drift in practice.

## Implementation Phases

1. **Crate scaffolding.** Add `crates/slac/` to the workspace members. Implement `Policy`, `Authorized<P>`, the `Any` / `All` / `from_fn` combinators, and the `policy!` macro. Cover with unit tests using stub policies.
1. **Auth foundation in `terrier-server`.** Add `crates/terrier-server/src/auth/` with the `axum-oidc` wiring, the `CurrentUser` extractor, and the atomic `Policy` implementations justified by the entities in RFC 0008.
1. **First worked example.** Convert a single route end-to-end (e.g. `PATCH /h/{slug}/settings`) to use `Authorized<SettingsAccess>` so that future handlers have a concrete template.
1. **Per-domain rollout.** Add new routes domain-by-domain (registration, judging, teams, schedule, results) using the pattern.
1. **External adoption.** Promote SLAC to its own repository or to crates.io once a second ScottyLabs service is ready to depend on it.
